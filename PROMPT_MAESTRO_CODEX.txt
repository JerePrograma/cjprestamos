Quiero que trabajes sobre este repositorio como agente de implementación para un sistema web interno de préstamos.

CONTEXTO FUNCIONAL
- El sistema NO es una fintech ni una plataforma de cobranza automatizada.
- El sistema está pensado para uso interno, con una sola operadora principal.
- El objetivo principal es mejorar la organización y resolver cuentas matemáticas simples.
- El sistema debe permitir registrar personas conocidas, legajos, préstamos, cuotas, pagos, referencias y métricas generales.
- El enfoque es manual-first: el sistema puede sugerir cálculos, pero la operadora mantiene control manual sobre decisiones y ajustes.
- El sistema debe sentirse como una libreta digital ordenada con cálculo asistido, no como una herramienta financiera rígida.

REGLAS DE NEGOCIO CLAVE
- Los préstamos se otorgan principalmente a conocidos.
- El préstamo se entrega siempre en un solo pago.
- El interés no es fijo por política general; depende de la persona y del caso.
- El sistema debe sugerir cálculo por porcentaje fijo, pero la decisión final puede ajustarse manualmente.
- Las cuotas pueden ser fijas o variar según el caso.
- Las fechas pueden ser automáticas o manuales.
- Puede haber pagos parciales, pagos múltiples o pagos adelantados.
- Adelantar pagos no implica necesariamente quitar interés.
- No hay que automatizar punitorios ni recordatorios en el MVP.
- La operadora quiere poder anotar referencias personales, contexto del cliente y datos útiles para decidir.
- Debe existir separación entre préstamos actuales y legajo.

PRIORIDADES DEL MVP
1. Registrar persona
2. Registrar monto inicial
3. Registrar cantidad de cuotas y montos
4. Calcular total, ganancia y monto por ganar
5. Registrar pagos
6. Ver deuda total
7. Mantener referencias y legajo separados

NO PRIORIZAR EN EL MVP
- Recordatorios automáticos
- Mensajes automáticos para WhatsApp
- Portal cliente
- Pagarés o recibos complejos
- Scoring financiero avanzado
- Notificaciones automáticas
- Automatizaciones que la operadora no pidió

REGLAS DE IMPLEMENTACIÓN
- Todo el dominio y los nombres funcionales deben estar en español.
- Mantené el código simple, claro y sin sobreingeniería.
- No introduzcas patrones complejos si no hay una razón fuerte.
- No agregues funcionalidades no pedidas.
- Separá claramente DTOs, entidades, servicios y controladores.
- Toda regla matemática debe estar encapsulada en servicios testeables.
- Priorizá exactitud matemática y consistencia por encima de sofisticación visual.
- Si una decisión funcional no es obvia, preferí la opción más simple y explícita.
- No mezcles préstamos activos con legajo personal.
- Las referencias personales no deben transformarse en un sistema de códigos crípticos.

REGLAS DE TRABAJO
- Antes de hacer cambios, leé el código existente y entendé la estructura del proyecto.
- Limitate a la tarea puntual que te indique.
- No refactorices partes no relacionadas salvo que sea necesario para completar la tarea.
- Si modificás archivos, mantené consistencia de estilo con el código existente.
- Si agregás endpoints, agregá DTOs, validaciones y tests.
- Si agregás lógica de negocio, agregá tests unitarios.
- Si agregás UI, integrala con la API real o dejá adaptadores claros.
- Si una tarea requiere migraciones de base de datos, usá Flyway.
- Si una tarea requiere persistencia, no dejes lógica crítica solamente en frontend.
- Documentá brevemente las decisiones técnicas importantes dentro del PR o resumen final.

DEFINICIÓN DE TERMINADO GENERAL
Una tarea se considera terminada si:
- compila
- los tests relevantes pasan
- respeta el alcance pedido
- no rompe funcionalidades existentes
- deja código legible
- deja el sistema más cerca del MVP real, no de una demo ficticia

MODO DE RESPUESTA ESPERADO
Cuando completes una tarea, quiero que indiques:
1. qué hiciste
2. qué archivos tocaste
3. qué decisiones tomaste
4. qué tests corriste
5. qué quedó pendiente o fuera de alcance

IMPORTANTE
No quiero humo, ni arquitectura ornamental, ni features inventadas.
Quiero avance real, incremental y verificable.
